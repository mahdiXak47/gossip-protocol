# مستندات پیاده‌سازی پروژه Gossip Protocol

این سند به‌زبان فارسی توضیح می‌دهد چه چیزی در این پروژه ساخته شده و چگونه ساخته شده است.

---

## ۱. خلاصهٔ پروژه

یک **شبکهٔ هم‌تا-به-هم‌تا (P2P)** بر اساس **پروتکل گاسپ (Gossip)** پیاده‌سازی شده است. هر نود (node) یک فرایند مستقل است که با نودهای دیگر از طریق **UDP** و **پیام‌های JSON** ارتباط برقرار می‌کند. هیچ سرور مرکزی وجود ندارد؛ نودها فقط با یک یا چند **نود bootstrap** برای ورود اولیه به شبکه آشنا می‌شوند و پس از آن به‌صورت کاملاً غیرمتمرکز عمل می‌کنند.

**هدف:** انتشار اطلاعات (مثلاً یک رویداد یا خبر) در شبکه به‌گونه‌ای که هر نود هر پیام را حداکثر **یک بار** پردازش کند و پیام در کل شبکه پخش شود.

---

## ۲. آنچه ساخته شده است

### ۲.۱ دو برنامهٔ اصلی

| برنامه | مسیر ساخت | کاربرد |
|--------|------------|--------|
| **node** | `./cmd/node` | یک نود گاسپ: گوش دادن روی UDP، مدیریت همسایه‌ها، دریافت و ارسال گاسپ، و امکان ورود اطلاعات از stdin. |
| **simulate** | `./cmd/simulate` | اجرای خودکار آزمایش: راه‌اندازی N نود، تزریق یک پیام گاسپ، و محاسبهٔ زمان همگرایی و تعداد پیام‌های ارسالی. |

### ۲.۲ قابلیت‌های نود

- **گوش دادن روی UDP:** دریافت پیام‌های JSON (HELLO، GET_PEERS، PEERS_LIST، GOSSIP، PING، PONG).
- **اتصال به شبکه (bootstrap):** ارسال HELLO و GET_PEERS به یک آدرس bootstrap و دریافت لیست همسایه‌ها (PEERS_LIST).
- **مدیریت همسایه‌ها:** نگه‌داری لیست همسایه‌ها با حداقل `peer_limit`، به‌روزرسانی `last_seen_at` با PING/PONG، و حذف نودهای مرده (timeout).
- **پخش گاسپ:** با ورود `gossip <topic> <data>` از stdin، ساخت یک پیام گاسپ با `msg_id` یکتا و ارسال به `fanout` همسایهٔ تصادفی؛ جلوگیری از پردازش تکراری با **seen-set** و **TTL**.
- **پارامترهای قابل تنظیم:** fanout، ttl، peer_limit، ping_interval، peer_timeout، seed (برای رفتار قطعی).
- **لاگ ساخت‌یافته:** رویدادهای مهم (startup، peer_add، peer_remove، gossip_receive، gossip_forward، duplicate، ping/pong، message_sent) به‌صورت JSON در هر خط (برای تحلیل و شبیه‌سازی).

### ۲.۳ قابلیت‌های شبیه‌ساز

- راه‌اندازی همزمان N نود (مثلاً ۱۰، ۲۰، ۵۰) با پورت‌های متوالی.
- نود اول به‌عنوان bootstrap (بدون `-bootstrap`)؛ بقیه با `-bootstrap` به آن متصل می‌شوند.
- تزریق یک پیام گاسپ (مثلاً `gossip sim run-0`) از stdin نود شمارهٔ ۱.
- محاسبهٔ **زمان همگرایی:** فاصلهٔ زمانی تا وقتی ۹۵٪ نودها آن پیام را دریافت کرده‌اند.
- محاسبهٔ **overhead پیام:** تعداد کل ارسال‌های GOSSIP برای آن پیام.
- خروجی: میانگین و انحراف معیار روی چند run با seedهای مختلف.

## ۳. معماری و نحوهٔ ساخت

### ۳.۱ ساختار کلی کد

```
gossip protocol/
├── docs/                    # مستندات (انگلیسی و فارسی)
├── internal/
│   ├── protocol/            # تعریف پروتکل و وضعیت نود
│   │   ├── types.go        # MsgType، Config، PeerEntry
│   │   ├── message.go      # پوشش پیام (Envelope) و payloadها
│   │   └── node_state.go   # وضعیت نود، لیست همسایه، seen-set، توابع مدیریت همسایه
│   ├── transport/           # ارسال/دریافت روی UDP
│   │   └── udp.go           # Send، SendToAddr، Receive (JSON)
│   └── logger/              # لاگ ساخت‌یافته برای تحلیل
│       └── logger.go        # رویدادها به‌صورت JSON در هر خط
├── cmd/
│   ├── node/                 # برنامهٔ نود
│   │   └── main.go           # پارامترها، bootstrap، حلقهٔ دریافت، گاسپ از stdin، PING، حذف نود مرده
│   └── simulate/             # برنامهٔ شبیه‌سازی
│       └── main.go           # راه‌اندازی N نود، تزریق گاسپ، پارس لاگ، محاسبهٔ همگرایی و overhead
├── go.mod
├── node                      # باینری نود (خروجی build)
└── simulate                  # باینری شبیه‌ساز (خروجی build)
```

### ۳.۲ لایهٔ پروتکل (`internal/protocol`)

- **types.go:** نوع پیام (HELLO، GET_PEERS، PEERS_LIST، GOSSIP، PING، PONG)، ساختار Config (fanout، ttl، peer_limit، ping_interval، peer_timeout)، و ساختار هر همسایه (PeerEntry: node_id، addr، last_seen_at، pending_pings).
- **message.go:** پوشش مشترک همهٔ پیام‌ها (Envelope) با فیلدهای version، msg_id، msg_type، sender_id، sender_addr، timestamp_ms، ttl، payload؛ به‌اضافهٔ ساختار payload برای هر نوع پیام.
- **node_state.go:** نگه‌داری وضعیت نود (node_id، self_addr، config، peers، seen-set). توابع: افزودن/حذف همسایه، به‌روزرسانی last_seen، انتخاب fanout همسایهٔ تصادفی، تشخیص همسایه‌های قدیمی (برای حذف به‌خاطر timeout). همهٔ دسترسی‌ها با قفل (mutex) برای امنیت در محیط چندنخی.

### ۳.۳ لایهٔ انتقال (`internal/transport`)

- **udp.go:** سریال‌سازی Envelope به JSON و ارسال به یک آدرس UDP؛ خواندن یک بستهٔ UDP و جداسازی JSON به Envelope. خطای JSON باعث crash نمی‌شود؛ در حلقهٔ دریافت نود خطا لاگ شده و ادامه داده می‌شود.

### ۳.۴ لایهٔ لاگ (`internal/logger`)

- **logger.go:** هر رویداد به‌صورت یک شیء JSON در یک خط (t_ms، node، event، payload). نام رویدادها ثابت است (مثلاً startup، peer_add، gossip_receive، duplicate، message_sent) تا اسکریپت‌های تحلیل و شبیه‌ساز بتوانند آن‌ها را پارس کنند.

### ۳.۵ برنامهٔ نود (`cmd/node/main.go`)

- **ورودی:** port، bootstrap (اختیاری)، fanout، ttl، peer-limit، ping-interval، peer-timeout، seed، logfile (اختیاری).
- **راه‌اندازی:** ساخت NodeState، اتصال UDP به `127.0.0.1:port`. اگر bootstrap داده شده باشد، یک‌بار فراخوانی `bootstrapJoin` (HELLO → GET_PEERS → انتظار PEERS_LIST و پر کردن لیست همسایه‌ها).
- **همزمانی:**
  - **گوروتین stdin:** خواندن خط از stdin؛ اگر با `gossip` شروع شود، ساخت پیام گاسپ و ارسال به fanout همسایه.
  - **گوروتین PING:** هر ping_interval به همهٔ همسایه‌ها PING ارسال و افزایش pending_pings.
  - **گوروتین حذف نود مرده:** هر چند ثانیه همسایه‌هایی که last_seen_at قدیمی‌تر از peer_timeout است حذف می‌شوند.
  - **حلقهٔ اصلی:** دریافت پیام از UDP و بر اساس msg_type واکنش (HELLO → افزودن فرستنده به همسایه؛ GET_PEERS → پاسخ PEERS_LIST؛ PING → پاسخ PONG؛ GOSSIP → اگر msg_id در seen نبود، اضافه به seen، کاهش TTL، در صورت TTL>0 ارسال به fanout همسایه؛ و غیره).
- **نود init:** هیچ کد جداگانه‌ای ندارد. نود init همان باینری `node` است که **بدون** پرچم `-bootstrap` اجرا شده است. وقتی نود دیگری GET_PEERS به آن بفرستد، در همین حلقهٔ دریافت پاسخ PEERS_LIST می‌دهد.

### ۳.۶ برنامهٔ شبیه‌ساز (`cmd/simulate/main.go`)

- برای هر run: ساخت N فرایند `node` با پورت‌های base_port تا base_port+N-1؛ نود ۰ بدون bootstrap، بقیه با bootstrap به نود ۰. برای نود ۱ یک StdinPipe گرفته می‌شود تا بعد از settle زمان، رشتهٔ `gossip sim run-<r>` به stdin نوشته شود.
- پس از run_dur، همهٔ نودها kill می‌شوند و فایل‌های لاگ در `logdir` خوانده می‌شوند. با پیدا کردن رویداد gossip_origin با topic=sim، msg_id آن پیام استخراج می‌شود؛ سپس همهٔ gossip_receive و message_sent با آن msg_id شمارش و زمان‌ها برای محاسبهٔ همگرایی (۹۵٪ نودها) و overhead استفاده می‌شوند.
- در پایان میانگین و انحراف معیار convergence_ms و message_overhead روی runها چاپ می‌شود.

## ۴. مفاهیم اصلی و طراحی

- **گاسپ push:** نود اطلاعات را به چند همسایهٔ تصادفی می‌فرستد؛ آن‌ها هم به همسایه‌های خود می‌فرستند. پخش بدون سرور مرکزی انجام می‌شود.
- **جلوگیری از تکرار:** هر پیام یک `msg_id` یکتا دارد؛ هر نود یک مجموعهٔ seen از msg_idهای دیده‌شده نگه می‌دارد. اگر msg_id قبلاً دیده شده باشد، پیام نادیده گرفته می‌شود.
- **TTL:** با هر بار ارسال گاسپ، TTL یکی کم می‌شود؛ وقتی به صفر برسد، ارسال متوقف می‌شود تا پیام بی‌نهایت دور نزند.
- **Fanout:** تعداد همسایه‌هایی که هر نود برای هر گاسپ به‌صورت تصادفی انتخاب می‌کند (محدود کردن پهنای ارسال و تنوع مسیر).
- **Bootstrap / نود init:** فقط یک نقطهٔ ورود برای نودهای تازه است. نود جدید با یک آدرس bootstrap شروع می‌کند، HELLO و GET_PEERS می‌فرستد و PEERS_LIST می‌گیرد. پس از آن مانند هر نود دیگر رفتار می‌کند؛ نود init کنترلر مرکزی نیست و همان کد نود عادی را اجرا می‌کند.

---

## ۵. جمع‌بندی

در این پروژه یک **شبکهٔ P2P گاسپ** با **زبان Go** پیاده‌سازی شده است: تعریف پروتکل و وضعیت نود در `internal/protocol`، ارسال/دریافت UDP در `internal/transport`، لاگ ساخت‌یافته در `internal/logger`، و دو برنامهٔ `node` (برای اجرای واقعی نودها) و `simulate` (برای آزمایش خودکار و اندازه‌گیری همگرایی و overhead). نود init به‌صورت جدا پیاده‌سازی نشده؛ همان نود بدون `-bootstrap` است که دیگران به آن متصل می‌شوند. برای جزئیات پروتکل و راهنمای استفاده به فایل‌های PROTOCOL_DESIGN.md و USER_GUIDE.md مراجعه شود.
