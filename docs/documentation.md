# مستند پیاده‌سازی Gossip Protocol

## معرفی پروژه

در این پروژه یک شبکه Gossip به زبان Go پیاده‌سازی شده است. هر نود از طریق UDP با نودهای دیگر ارتباط برقرار می‌کند و پیام‌ها را به صورت گاسپ در شبکه پخش می‌کند.

هر پیام در هر نود حداکثر یک بار پردازش می‌شود. این کار با استفاده از msg_id و مجموعه seen انجام می‌شود.

ارتباط بین نودها بر پایه UDP است و پیام‌ها به صورت JSON ارسال می‌شوند.

پروژه شامل دو برنامه اصلی است:

- node  
اجرای یک نود در شبکه Gossip

- simulate  
اجرای آزمایش‌های خودکار روی شبکه و اندازه‌گیری عملکرد انتشار گاسپ

---

## معماری کلی

شبکه به صورت P2P کار می‌کند و هیچ سرور مرکزی وجود ندارد. هر نود فقط همسایه‌های خودش را می‌شناسد و پیام‌ها را به صورت تصادفی بین همسایه‌ها پخش می‌کند.

برای ورود نودهای جدید به شبکه از bootstrap node استفاده می‌شود. bootstrap فقط نقطه شروع اتصال است و نقش کنترلی ندارد.

---

## نود bootstrap

در این پروژه برنامه جداگانه‌ای برای bootstrap وجود ندارد. bootstrap همان برنامه node است که بدون پارامتر bootstrap اجرا می‌شود.

مثال:

```
./node -port 9000
```

نودهای دیگر می‌توانند با استفاده از آدرس bootstrap وارد شبکه شوند:

```
./node -port 8000 -bootstrap 127.0.0.1:9000
```

هر نود فقط یک آدرس bootstrap می‌گیرد. در صورت نیاز می‌توان چند bootstrap node اجرا کرد.

نود bootstrap رفتار خاصی ندارد. فقط وقتی پیام GET_PEERS دریافت کند، لیست همسایه‌های فعلی خودش را برمی‌گرداند.

---

## وضعیت داخلی نود

هر نود اطلاعات زیر را نگه می‌دارد:

| فیلد | توضیح |
|-----|------|
| node_id | شناسه یکتا برای نود |
| self_addr | آدرس نود به صورت IP:PORT |
| peers | لیست همسایه‌ها |
| seen | مجموعه msg_id های دیده شده |
| config | تنظیمات نود |

هر همسایه به صورت زیر ذخیره می‌شود:

| فیلد | توضیح |
|-----|------|
| node_id | شناسه همسایه |
| addr | آدرس همسایه |
| last_seen_at | آخرین زمان دریافت پاسخ |
| pending_pings | تعداد pingهای بی‌پاسخ |

---

## تنظیمات نود

پارامترهای زیر هنگام اجرای نود قابل تنظیم هستند:

- fanout  
تعداد همسایه‌هایی که هر گاسپ به آن‌ها ارسال می‌شود

- ttl  
حداکثر تعداد hop برای هر گاسپ

- peer-limit  
حداکثر تعداد همسایه‌ها

- ping-interval  
فاصله ارسال ping

- peer-timeout  
زمانی که بعد از آن همسایه حذف می‌شود

- seed  
برای رفتار تصادفی قابل تکرار

---

## فرمت پیام‌ها

تمام پیام‌ها یک ساختار مشترک دارند:

```json
{
  "version": 1,
  "msg_id": "uuid",
  "msg_type": "HELLO | GET_PEERS | PEERS_LIST | GOSSIP | PING | PONG",
  "sender_id": "node-id",
  "sender_addr": "127.0.0.1:8000",
  "timestamp_ms": 123456,
  "ttl": 8,
  "payload": {}
}
```

msg_id برای جلوگیری از پردازش تکراری استفاده می‌شود.

ttl فقط برای پیام‌های GOSSIP استفاده می‌شود.

---

## انواع پیام‌ها

HELLO

```
{"capabilities":["udp","json"]}
```

GET_PEERS

```
{"max_peers":20}
```

PEERS_LIST

```
{"peers":[
  {"node_id":"...","addr":"127.0.0.1:8001"}
]}
```

GOSSIP

```
{
 "topic":"news",
 "data":"hello",
 "origin_id":"node-id",
 "origin_timestamp_ms":12345
}
```

PING

```
{"ping_id":"id","seq":1}
```

PONG

```
{"ping_id":"id","seq":1}
```

## پیوستن به شبکه

وقتی یک نود جدید اجرا می‌شود:

1. آدرس bootstrap را دارد
2. پیام HELLO ارسال می‌کند
3. پیام GET_PEERS ارسال می‌کند
4. پیام PEERS_LIST دریافت می‌کند
5. لیست peers را پر می‌کند
6. شروع به ارسال ping و دریافت گاسپ می‌کند

بعد از join شدن، bootstrap دیگر نقش خاصی ندارد.

---

## مدیریت همسایه‌ها

هر نود به صورت دوره‌ای به همسایه‌ها PING ارسال می‌کند.

اگر همسایه PONG برگرداند مقدار last_seen_at به‌روز می‌شود.

اگر همسایه تا مدت peer-timeout پاسخی ندهد حذف می‌شود.

اگر تعداد peers به peer-limit برسد و همسایه جدیدی اضافه شود، یکی از همسایه‌های قدیمی حذف می‌شود.

---

## جلوگیری از تکرار گاسپ

هر پیام یک msg_id یکتا دارد.

هر نود مجموعه seen دارد.

وقتی پیام GOSSIP دریافت شود:

1. اگر msg_id قبلاً دیده شده باشد پیام نادیده گرفته می‌شود
2. اگر جدید باشد:
   - msg_id ذخیره می‌شود
   - ttl کم می‌شود
   - اگر ttl بزرگتر از صفر باشد پیام به fanout همسایه تصادفی ارسال می‌شود

---

## اجرای نود

نمونه اجرا:

```
./node \
-port 8000 \
-bootstrap 127.0.0.1:9000 \
-fanout 3 \
-ttl 8 \
-peer-limit 20 \
-ping-interval 2s \
-peer-timeout 6s \
-seed 42
```

ارسال گاسپ:

```
gossip news hello
```

ذخیره لاگ:

```
-logfile node.log
```

---

## شبیه‌ساز

برنامه simulate برای اجرای آزمایش‌ها استفاده می‌شود.

در هر run تعداد N نود اجرا می‌شود.

نود اول bootstrap است.

بقیه نودها به آن متصل می‌شوند.

بعد از مدتی یک پیام گاسپ از نود شماره ۱ ارسال می‌شود.

در پایان لاگ‌ها خوانده می‌شوند و دو مقدار محاسبه می‌شود:

زمان همگرایی  
مدت زمانی که طول می‌کشد تا ۹۵٪ نودها پیام را دریافت کنند.

تعداد پیام‌ها  
تعداد کل پیام‌های GOSSIP ارسال شده.

در پایان میانگین و انحراف معیار چاپ می‌شود.

## Proof of Work و Sybil

برای جلوگیری از ایجاد تعداد زیاد نود جعلی می‌توان در پیام HELLO یک PoW قرار داد.

در این حالت نود باید یک nonce پیدا کند که هش HELLO همراه nonce دارای k بیت صفر در ابتدای خروجی باشد.

در نتیجه ساخت هر نود جدید نیاز به محاسبه دارد و حمله Sybil پرهزینه‌تر می‌شود.

اگر k کوچک باشد تاثیر کمی دارد.

اگر k بزرگ باشد زمان join زیاد می‌شود.

مقادیر حدود 4 یا 5 معمولاً انتخاب مناسبی هستند، ولی بهتر است با آزمایش روی سیستم واقعی تعیین شوند.
